**Identifying resource requirements** is the process of determining exactly what resources—and in what quantities—are needed to successfully complete a project before any allocation or scheduling takes place.

It answers the question: _"What do we need to get this project done?"_

---
## The Steps Explained

| Step                                       | What It Means                                            | Short Explanation                                                                                      |
| ------------------------------------------ | -------------------------------------------------------- | ------------------------------------------------------------------------------------------------------ |
| **1. Define project objectives and scope** | Clarify what the project must achieve and its boundaries | Without a clear scope, you cannot determine what resources are needed. This sets the "what" and "why." |
| **2. Break down the project into tasks**   | Decompose work into manageable units (WBS)               | Creates the granular list of activities that will eventually consume resources.                        |
| **3. Conduct task analysis**               | Understand what each task requires                       | Identifies dependencies, prerequisites, and the nature of work (e.g., coding, testing, design).        |
| **4. Determine skill levels**              | Specify expertise needed per task                        | Distinguishes between junior, mid-level, and senior talent requirements.                               |
| **5. Estimate effort and duration**        | Quantify hours and calendar time                         | Converts tasks into measurable units like person-hours, story points, or sprint capacity.              |
| **6. Identify roles and responsibilities** | Define who is accountable for what                       | Clarifies ownership using tools like RACI to prevent ambiguity and overlap.                            |
| **7. Consider resource constraints**       | Acknowledge limitations                                  | Includes budget caps, limited availability of specialists, or hardware procurement delays.             |
| **8. Assess hardware and software needs**  | Identify technical infrastructure                        | Lists specific tools, licenses, cloud instances, and test devices required.                            |
| **9. Factor in contingencies**             | Add buffer for uncertainty                               | Accounts for risks like sick leave, scope creep, or vendor delays (typically 10–20% buffer).           |
| **10. Evaluate existing resources**        | Inventory what is already available                      | Identifies internal talent, existing licenses, or reusable code that can be leveraged.                 |
| **11. Perform gap analysis**               | Compare required vs. available                           | Highlights shortages (e.g., "need 2 senior backend devs, have 0") that must be filled.                 |
| **12. Consider workload availability**     | Check current allocation of team members                 | Ensures people are not already overcommitted to other projects before assignment.                      |
| **13. Review historical data**             | Learn from past projects                                 | Uses benchmarks from similar past work to validate estimates and avoid repeating mistakes.             |

---

## Non-Negotiable Steps (The Essential Core)

While all steps add value, **five are absolutely critical**. If any of these are skipped, resource planning becomes guesswork and the project is highly likely to face failure.

### 1. Define Project Objectives and Scope
> **Why it's non-negotiable:** Resources cannot be identified for an undefined goal. Scope ambiguity guarantees either overallocation (waste) or underallocation (missed deadlines).

- Without this, you don't know what you're resourcing *for*.

### 2. Break Down the Project into Tasks (WBS)
> **Why it's non-negotiable:** High-level estimates are consistently wrong. You cannot accurately identify resources without a granular view of the actual work.

- A work breakdown structure (WBS) is the foundation upon which all effort, skill, and duration estimates are built.

### 3. Estimate Effort and Duration
> **Why it's non-negotiable:** This converts tasks into quantifiable resource demand. If you cannot estimate how much time a task takes, you cannot determine how many people or how much money is needed.

- Effort drives staffing. Duration drives scheduling. Both are essential.

### 4. Perform Gap Analysis
> **Why it's non-negotiable:** Identifying what you *need* is useless if you don't compare it to what you *have*. The gap tells you exactly what to hire, buy, or reallocate.

- This step transforms a theoretical resource list into an actionable hiring or procurement plan.

### 5. Consider Resource Constraints
> **Why it's non-negotiable:** An ideal resource plan that ignores constraints (budget, availability, skill scarcity) is a fantasy. Reality forces trade-offs.

- Constraints define feasibility. Without acknowledging them early, the plan will fail during execution.

## Why These Five Are the Backbone

| Non-Negotiable Step | The Question It Answers |
|---------------------|-------------------------|
| Define scope | What are we building? |
| Break down tasks | What specific work needs to happen? |
| Estimate effort/duration | How much work is there, and how long will it take? |
| Gap analysis | What do we lack that we must acquire? |
| Consider constraints | Is this even possible given our limits? |

The remaining steps (skill levels, roles, hardware assessment, contingencies, historical data, workload availability) are **highly important** for accuracy and risk management, but they function as refinements to the core five. If you have the core five right, you can recover from errors in the others. If you miss any of the core five, no amount of detail in the rest will save the plan.
