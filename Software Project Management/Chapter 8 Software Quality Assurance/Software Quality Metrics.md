
### 1. Correctness

**Definition:** Conformance to the specifications and user requirements.

**What it measures:** How well the software does exactly what it is supposed to do according to requirements.

**Example metric:**  
- Percentage of requirements fulfilled  
- Number of functional defects found

**Question answered:** *Does the software meet its specifications?*

---

### 2. Reliability

**Definition:** Performing the expected function with required precision.

**What it measures:** How consistently the software performs without failure over time.

**Example metrics:**  
- Mean Time Between Failures (MTBF)  
- Failure rate per hour of operation

**Question answered:** *How often does the software fail?*

---

### 3. Efficiency

**Definition:** Number of resources required to execute the required function.

**What it measures:** How well the software uses system resources (CPU, memory, network, battery).

**Example metrics:**  
- Response time  
- Memory usage  
- CPU utilization  
- Throughput (transactions per second)

**Question answered:** *Does the software use resources wisely?*

---

### 4. Integrity

**Definition:** Security features or security controls implemented in the software.

**What it measures:** How well the software protects against unauthorized access, data breaches, and attacks.

**Example metrics:**  
- Number of security vulnerabilities  
- Time to patch a security flaw  
- Successful penetration test results

**Question answered:** *Is the software secure?*

---

### 5. Usability

**Definition:** Effort required to understand, learn, and operate the software.

**What it measures:** How easy and intuitive the software is for end users.

**Example metrics:**  
- Time to complete a task  
- User satisfaction score (e.g., System Usability Scale)  
- Number of user errors  
- Learning time for new users

**Question answered:** *Is the software easy to use?*

---

### 6. Maintainability

**Definition:** Effort required to maintain the software.

**What it measures:** How easy it is to fix bugs, add features, or modify the software.

**Example metrics:**  
- Cyclomatic complexity (lower is better)  
- Code duplication percentage  
- Time to fix a defect  
- Code comment density

**Question answered:** *How easy is it to change the software?*

---

### 7. Testability

**Definition:** Effort required to test the software.

**What it measures:** How easy it is to write and execute tests, and to isolate defects.

**Example metrics:**  
- Number of test cases per function  
- Time to achieve test coverage  
- Availability of test hooks or APIs

**Question answered:** *How easy is it to test the software?*

---

### 8. Flexibility

**Definition:** Effort required to make changes to the software.

**What it measures:** How adaptable the software is to future changes (new features, different environments).

**Example metrics:**  
- Time to implement a new feature  
- Number of modules affected by a change  
- Coupling between modules (lower is better)

**Question answered:** *How easy is it to adapt the software to new requirements?*

---

### 9. Portability

**Definition:** Effort required to port software from one platform or configuration to another.

**What it measures:** How easily the software can be moved to different operating systems, browsers, hardware, or environments.

**Example metrics:**  
- Number of platform-dependent lines of code  
- Installation success rate on target platforms  
- Time to deploy on a new environment

**Question answered:** *Can the software run on different platforms?*

---

### 10. Interoperability

**Definition:** Effort required to couple systems with one another.

**What it measures:** How easily the software can exchange data and work with other systems, APIs, or third-party services.

**Example metrics:**  
- Number of successful API integrations  
- Time to integrate with a standard protocol  
- Adherence to data format standards (XML, JSON, etc.)

**Question answered:** *Does the software work well with other systems?*
