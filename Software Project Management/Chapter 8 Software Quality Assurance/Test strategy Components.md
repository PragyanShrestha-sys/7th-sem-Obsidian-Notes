## What a Test Strategy Contains

A test strategy is a high-level document that defines the overall approach to testing. It includes the following 8 components:

| # | Component | What It Means |
|---|-----------|----------------|
| **1** | **Scope and Objective** | Defines the business objective and how much testing scope is included. |
| **2** | **Business Issues** | Covers budget, time required for testing, resources needed, and other business constraints. |
| **3** | **Testing Approach** | Specifies types of testing (performance, load, stress, functional, etc.) and whether testing is manual, automated, or both. |
| **4** | **Test Deliverables** | Lists documents required from the testing team and how testing records will be maintained. |
| **5** | **Defect Tracking Approach** | Defines which tool will be used for tracking defects, how the testing team will communicate with development, and the defect flow process. |
| **6** | **Training** | Identifies what training is needed (e.g., for new or complex tools) and who is responsible for conducting it. |
| **7** | **Automation** | Plans automation details: script language, tool, reporting, and code maintenance (if automation is needed). |
| **8** | **Risks** | Identifies obvious risks that can be anticipated and includes possible solutions if those risks occur. |

---

## Summary

> A test strategy answers **"What is our high-level testing plan?"** by covering:  
> **Scope, Budget/Resources, Testing Types, Deliverables, Defect Tracking, Training, Automation, and Risks.**
