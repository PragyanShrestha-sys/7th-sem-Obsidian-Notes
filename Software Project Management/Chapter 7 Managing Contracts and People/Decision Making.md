
## Types of Decisions

Decisions can be broadly classified into **two categories**:

### 1. Structured Decisions

**Definition:** Routine and simple decisions that follow **predefined rules and procedures**.

| Aspect | Description |
|--------|-------------|
| **Characteristics** | - Well-defined problems <br>- Clear rules and guidelines available <br>- Easy to automate or standardize |
| **Examples** | - Assigning routine tasks <br>- Following standard procedures (e.g., approving a leave request, allocating a bug fix to the available developer) |

**In software projects:** Deciding which developer gets the next ticket from a prioritized backlog (if a clear rule like "lowest workload" exists).

---

### 2. Unstructured Decisions

**Definition:** Complex and **non-routine decisions that require judgment and creativity**.

| Aspect | Description |
|--------|-------------|
| **Characteristics** | - No clear rules or procedures <br>- High level of uncertainty <br>- Requires experience and innovative thinking |
| **Examples** | - Designing a new software system (e.g., choosing between microservices vs. monolith) <br>- Solving unexpected project issues (e.g., a critical production outage with no known fix) |

**In software projects:** Deciding whether to refactor legacy code now or delay it to meet a release deadline—no procedure covers every unique context.

---

## Comparison Table

| Feature              | Structured Decisions           | Unstructured Decisions                      |
| -------------------- | ------------------------------ | ------------------------------------------- |
| **Nature**           | Routine, repetitive            | Unique, non-routine                         |
| **Rules**            | Predefined, clear              | No clear rules                              |
| **Uncertainty**      | Low                            | High                                        |
| **Approach**         | Follow procedure               | Judgment, creativity, experience            |
| **Automation**       | Easy to automate               | Difficult or impossible to automate         |
| **Software Example** | Assigning a daily standup time | Choosing a cloud provider for a new product |

---

## Practical Implication for Software Project Managers

- **Use structured decision-making** for routine operations (e.g., resource allocation, status reporting). Automate where possible.
- **Apply unstructured decision-making** for strategic or novel problems (e.g., technology selection, crisis response). Rely on experienced team input and collaborative problem-solving.

> **Takeaway:** Knowing which type of decision you face helps you choose the right approach—follow rules for structured problems, and engage creativity and judgment for unstructured ones.
